Past-Date Scheduling Enforcement

Engineering change summary — CatchAll AI

Date: March 18, 2026 Area: Social Calendar / Post Scheduling Type: Bug fix + validation Files changed: Multiple (see PR) New dependencies: None

Overview

Users were able to submit post scheduling forms with dates in the past, resulting in posts that appeared immediately overdue on the calendar. This change closes those gaps by enforcing future-date validation on both the frontend (input constraints + submit-time guard) and backend (server-side rejection with a 422 response).

ℹ
Session context: A prior session focused on CalendarPostModal, SchedulePostModal, BulkScheduleModal, BusinessReviewScheduler, and scheduleBusinessReview.ts. This session completes the remaining gaps identified by a follow-up audit and ensures consistent past-date prevention across the stack.

Technical Approach

Frontend pattern (applied to all 3 UI components)

  1. Add a todayLocal() or toLocalISOString() helper at module level to derive the correct local date/time — avoiding the UTC offset bug in native .toISOString().
  2. Set min={todayLocal()} (or min={toLocalISOString()} for datetime-local inputs) so the browser picker itself blocks past selection.
  3. Add a submit-time guard that re-evaluates new Date() at click time — catching the edge case where a user sets a valid time then waits past it.
  4. Surface a clear inline error: "Scheduled time must be in the future."

Backend pattern

After the existing presence check, parse the incoming date+time into a Date object and compare against new Date() on the server. Return 422 with a descriptive message if invalid.

const scheduledAt = new Date(scheduledDate + 'T' + (scheduledTime || '10:00') + ':00');
if (isNaN(scheduledAt.getTime()) || scheduledAt <= new Date()) {
  return Response.json({ error: 'Scheduled time must be in the future' }, { status: 422 });
}
⚠
UTC offset bug (fixed in ComposePostModal): new Date().toISOString().slice(0, 16) returns UTC time, but datetime-local inputs interpret values as local time. In timezones west of UTC (e.g. America/New_York, UTC-5), this caused the minimum selectable time to be 5 hours earlier than it should be. Fixed by using toLocalISOString() which offsets by getTimezoneOffset() before slicing.

Edge Cases

Scenario Behaviour
User sets a valid time, then waits until it has passed before clicking submit The submit guard re-evaluates new Date() at click time — the stale value is caught and rejected.
User's local clock is wrong Frontend min/guard use the same clock so they stay self-consistent. The backend uses the server clock, providing a second line of defence for clients with drifted clocks.
ComposePostModal with empty scheduledTime Allowed — empty means save as draft, so no date validation is applied.
DraftFromAssetsModal (always creates as draft) Validated anyway — a draft with a past scheduled_date appears as overdue on the calendar, so we block it regardless of status.
Direct API call bypassing the frontend createDraftFromApproved.ts now validates server-side and rejects with 422. scheduleBusinessReview.ts was already protected (fixed in a prior session).

How to Verify

  1. BufferComposer: Open the Composer tab. Confirm the date picker minimum is today. Set a past date and click "Schedule Post" (non-draft) — expect inline error. Set a future date — expect success.
  2. DraftFromAssetsModal: Open the modal. Confirm date defaults to today. Attempt to create with a past date — expect error. Use a future date — expect the draft to be created.
  3. ComposePostModal: Open the Schedule tab. Confirm the datetime-local input's minimum reflects local time (not UTC). Set a past time and submit — expect error. Leave time blank — expect draft save to proceed.
  4. createDraftFromApproved (backend): POST with a past scheduledDate — expect 422 { error: "Scheduled time must be in the future" }. POST with a future date — expect 200.
